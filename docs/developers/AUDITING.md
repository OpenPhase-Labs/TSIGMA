## 1. Overview

TSIGMA tracks changes to traffic signal configuration data through a combination of database triggers, ORM mixins, HTTP middleware, and structured logging.

Auditing in TSIGMA falls into three categories:

| Category | Purpose | Mechanism |
|---|---|---|
| **Configuration Audit** | Track who changed signal/approach/detector config and when | Dialect-aware triggers with `changed_by` attribution |
| **Authentication Audit** | Track login, logout, failed attempts, lockouts | `auth_audit_log` table via `log_auth_event()` |
| **Request Audit** | Correlate HTTP requests with users and timing | Middleware with user context injection |

## 2. Current Implementation

### 2.1 Configuration Audit Triggers

Database triggers capture every INSERT, UPDATE, and DELETE on the `signal`, `approach`, and `detector` tables. The triggers are created in the initial Alembic migration (`c0f911213673_initial_schema.py`) using `DialectHelper.audit_trigger_sql()`.

**Audited tables and their audit targets:**

| Source Table | Audit Table | ID Columns Captured |
|---|---|---|
| `signal` | `signal_audit` | `signal_id` |
| `approach` | `approach_audit` | `approach_id`, `signal_id` |
| `detector` | `detector_audit` | `detector_id`, `approach_id` |

**Audit table schema** (same structure for all three):

```sql
CREATE TABLE signal_audit (
    audit_id    BIGSERIAL PRIMARY KEY,
    signal_id   TEXT NOT NULL,
    changed_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    changed_by  TEXT,          -- populated from app.current_user session variable
    operation   TEXT NOT NULL,  -- 'INSERT', 'UPDATE', or 'DELETE'
    old_values  JSONB,
    new_values  JSONB
);
```

**Dialect-aware trigger generation** (`tsigma/database/db.py` ŌĆö `DialectHelper`):

Triggers are generated by `DialectHelper.audit_trigger_sql()`, which produces the correct SQL for each supported database:

| Dialect | User Variable Mechanism | JSON Serialization | Trigger Structure |
|---|---|---|---|
| **PostgreSQL** | `current_setting('app.current_user', true)` | `to_jsonb(OLD)` / `to_jsonb(NEW)` | Single trigger function + trigger |
| **MS-SQL** | `SESSION_CONTEXT(N'current_user')` | `FOR JSON PATH, WITHOUT_ARRAY_WRAPPER` | Single trigger with INSERTED/DELETED pseudo-tables |
| **Oracle** | `SYS_CONTEXT('CLIENTCONTEXT', 'current_user')` | NULL (JSON not natively supported in trigger) | Single compound trigger |
| **MySQL** | `@app_user` session variable | `JSON_OBJECT()` | Three separate triggers (INSERT, UPDATE, DELETE) |

**Migration code** (end of `upgrade()` in `c0f911213673`):

```python
from tsigma.config import settings
from tsigma.database.db import DialectHelper

dialect = DialectHelper(settings.db_type)

for table, audit_table, id_cols in [
    ("signal", "signal_audit", ["signal_id"]),
    ("approach", "approach_audit", ["approach_id", "signal_id"]),
    ("detector", "detector_audit", ["detector_id", "approach_id"]),
]:
    for stmt in dialect.audit_trigger_sql(table, audit_table, id_cols):
        op.execute(sa.text(stmt))
```

**PostgreSQL trigger function** (generated for `signal`):

```sql
CREATE OR REPLACE FUNCTION audit_signal_changes()
RETURNS TRIGGER AS $$
DECLARE
    v_user TEXT;
BEGIN
    v_user := current_setting('app.current_user', true);
    IF (TG_OP = 'DELETE') THEN
        INSERT INTO signal_audit (signal_id, changed_by, operation, old_values)
        VALUES (OLD.signal_id, v_user, 'DELETE', to_jsonb(OLD));
        RETURN OLD;
    ELSIF (TG_OP = 'UPDATE') THEN
        INSERT INTO signal_audit (signal_id, changed_by, operation, old_values, new_values)
        VALUES (NEW.signal_id, v_user, 'UPDATE', to_jsonb(OLD), to_jsonb(NEW));
        RETURN NEW;
    ELSIF (TG_OP = 'INSERT') THEN
        INSERT INTO signal_audit (signal_id, changed_by, operation, new_values)
        VALUES (NEW.signal_id, v_user, 'INSERT', to_jsonb(NEW));
        RETURN NEW;
    END IF;
END;
$$ LANGUAGE plpgsql;
```

**API endpoint:**

```
GET /api/v1/signals/{signal_id}/audit?skip=0&limit=100
```

Returns audit records ordered by `changed_at DESC` with pagination. Returns 404 if the signal does not exist.

### 2.2 User Attribution via get_audited_session

The `get_audited_session` dependency (`tsigma/dependencies.py`) sets the current application user on the database transaction so that audit triggers can populate `changed_by`. It resolves the authenticated user from the session cookie and executes dialect-specific SQL via `DialectHelper.set_app_user_sql()`.

**Dialect-specific SET commands:**

| Dialect | SQL | Scope |
|---|---|---|
| **PostgreSQL** | `SET LOCAL app.current_user = :username` | Transaction-scoped (`SET LOCAL` resets on commit/rollback) |
| **MS-SQL** | `EXEC sp_set_session_context N'current_user', :username` | Session-scoped (resets when connection returns to pool) |
| **Oracle** | `BEGIN DBMS_SESSION.SET_CONTEXT('CLIENTCONTEXT', 'current_user', :username); END;` | Session-scoped |
| **MySQL** | `SET @app_user = :username` | Session-scoped |

PostgreSQL uses `SET LOCAL`, which is transaction-scoped and inherently safe with connection pooling. Other dialects use session-scoped variables; the value is set at the start of each request and is overwritten by the next request that reuses the connection.

**Implementation** (`tsigma/dependencies.py`):

```python
async def get_audited_session(
    request: Request,
    session: AsyncSession = Depends(get_session),
) -> AsyncSession:
    from .auth.sessions import BaseSessionStore
    from .config import settings

    store: BaseSessionStore | None = getattr(request.app.state, "session_store", None)
    username = None

    if store:
        session_id = request.cookies.get(settings.auth_cookie_name)
        if session_id:
            session_data = await store.get(session_id)
            if session_data:
                username = session_data.username

    if username:
        from .database.db import get_db_facade
        facade = get_db_facade()
        sql = facade.dialect.set_app_user_sql()
        await session.execute(text(sql), {"username": username})

    return session
```

Key points:

- `get_audited_session` wraps the base `get_session` dependency and adds user attribution.
- When no user is authenticated (e.g., system/scheduler operations), the variable is not set, and the trigger records `changed_by = NULL`.
- No changes required to individual route handlers beyond switching from `get_session` to `get_audited_session`.

**Endpoints using `get_audited_session`:**

| Module | File |
|---|---|
| Signals CRUD | `tsigma/api/v1/signals.py` |
| Approaches CRUD | `tsigma/api/v1/approaches.py` |
| Detectors CRUD | `tsigma/api/v1/detectors.py` |
| Generic CRUD factory | `tsigma/api/v1/crud_factory.py` |

All write endpoints on audited tables use `get_audited_session` instead of `get_session`.

### 2.3 Authentication Audit Logging

Authentication events are logged to the `auth_audit_log` table by `log_auth_event()` (`tsigma/auth/audit.py`).

**Table schema:**

```sql
CREATE TABLE auth_audit_log (
    id          BIGSERIAL PRIMARY KEY,
    event_type  TEXT NOT NULL,
    user_id     UUID,
    username    TEXT NOT NULL,
    ip_address  TEXT,
    user_agent  TEXT,
    timestamp   TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

**Recorded event types:**

| Event Type | Trigger Location | Description |
|---|---|---|
| `login` | `tsigma/auth/providers/local.py` | Successful authentication |
| `login_failed` | `tsigma/auth/providers/local.py` | Invalid credentials or inactive user |
| `lockout` | `tsigma/auth/providers/local.py` | Too many failed attempts (5 within 5 minutes) |
| `logout` | `tsigma/auth/router.py` | User-initiated session termination |

**Function signature:**

```python
async def log_auth_event(
    session: AsyncSession,
    event_type: str,          # "login", "logout", "login_failed", "lockout"
    username: str,            # username involved (even for failed attempts)
    request: Request,         # for IP address and user agent
    user_id: UUID | None = None,  # resolved user ID (None for failures)
) -> None
```

Each call creates an `AuthAuditLog` ORM instance, adds it to the session, and flushes. The record captures the client IP address and user agent from the request headers.

**Indexes:**

```sql
CREATE INDEX idx_auth_audit_user ON auth_audit_log (user_id, timestamp DESC);
CREATE INDEX idx_auth_audit_type ON auth_audit_log (event_type, timestamp DESC);
CREATE INDEX idx_auth_audit_time ON auth_audit_log (timestamp DESC);
```

### 2.4 Base Timestamp Tracking (TimestampMixin)

All core configuration models inherit `TimestampMixin` (`tsigma/models/base.py`), which adds `created_at` and `updated_at` columns:

```python
class TimestampMixin:
    created_at: Mapped[datetime] = mapped_column(
        TIMESTAMP(timezone=True), nullable=False, server_default=func.now(),
    )
    updated_at: Mapped[datetime] = mapped_column(
        TIMESTAMP(timezone=True), nullable=False, server_default=func.now(),
        onupdate=func.now(),
    )
```

**Models using TimestampMixin:**

| Model | Table |
|---|---|
| `Signal` | `signal` |
| `Approach` | `approach` |
| `Detector` | `detector` |
| `PollingCheckpoint` | `polling_checkpoint` |
| `AuthUser` | `auth_user` |

This provides basic "when was this row last touched" tracking but does not record what changed or who changed it. The audit triggers (Section 2.1) provide that detail.

### 2.5 Request Tracking Middleware

Three middleware classes in `tsigma/middleware.py` provide per-request observability:

| Middleware | Header | Purpose |
|---|---|---|
| `RequestIDMiddleware` | `X-Request-ID` | Assigns a UUID to every request, stored in `request.state.request_id` |
| `TimingMiddleware` | `X-Process-Time` | Measures wall-clock processing time in milliseconds |
| `LoggingMiddleware` | -- | Logs `[request_id] METHOD PATH -> STATUS_CODE` |

The request ID is available to all downstream code via `request.state.request_id` and is returned to the client in the response header for correlation with server logs.

---

### 2.6 Structured Logging

TSIGMA uses Python's `logging` module configured for structured JSON output (controlled by `TSIGMA_LOG_FORMAT`):

- **Timestamps**: UTC ISO-8601
- **Fields**: timestamp, level, logger name, message
- **Formats**: `json` (machine-readable) or `console` (human-readable)

## 3. Additional Audit Capabilities

### 3.1 User Context in Request Logs

`LoggingMiddleware` (`tsigma/middleware.py`) logs each request with its request ID, HTTP method, path, and response status code:

```python
class LoggingMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request: Request, call_next: Callable) -> Response:
        request_id = getattr(request.state, "request_id", "no-id")
        method = request.method
        path = request.url.path

        logger.info("[%s] %s %s", request_id, method, path)

        response = await call_next(request)

        logger.info("[%s] %s %s -> %d", request_id, method, path, response.status_code)

        return response
```

Log output format:

```
[abc-123] GET /api/v1/signals -> 200
```

User identity can be correlated with requests via the auth audit log and the request ID. The `get_audited_session` dependency sets the username on the database transaction for audit trigger attribution.

### 3.2 Audit Data Retention Policy

Audit data retention follows these defaults:

| Audit Type | Default Retention | Rationale |
|---|---|---|
| Configuration audit (`signal_audit`, `approach_audit`, `detector_audit`) | Indefinite | Compliance, change history |
| Authentication audit (`auth_audit_log`) | Indefinite | Security review |

There is no automated retention job. Manual cleanup can be performed via SQL or a custom `@JobRegistry.register` background job if storage becomes a concern.
